# WikiRAG — Local Wikipedia AI Assistant

## Overview

WikiRAG is a fully local, ChatGPT-style question answering system built for educational purposes. It ingests Wikipedia pages for 20 famous people and 20 notable places, chunks and embeds their content into a persistent ChromaDB vector database, and answers user questions with grounded, source-cited responses generated by a local Mistral 7B language model running through Ollama. No data ever leaves your machine — no OpenAI, no cloud APIs, no tracking.

The system is built entirely from scratch using custom chunking, custom retrieval, and direct HTTP calls to Ollama — without LangChain, LlamaIndex, or any other high-level RAG framework. This makes every component transparent, auditable, and educational. It is presented through a polished Streamlit GUI with a dark theme, real-time ingestion progress, a vector store explorer, a full chat interface with source citations, and a sidebar with quick question suggestions.

## Architecture

```
Ingestion Pipeline (run once):
─────────────────────────────────────────────────────────────────────
Wikipedia  →  Fetcher  →  Chunker  →  Embedder  →  ChromaDB
 (API)     (fetcher.py) (chunker.py) (embedder.py)  (chroma_store.py)
                                                          │
                                                   SQLite tracker
                                                (sqlite_tracker.py)

Query Pipeline (every user message):
─────────────────────────────────────────────────────────────────────
User Query  →  Classifier  →  Embedder  →  ChromaDB  →  LLM  →  Answer
            (classifier.py)              (retriever.py) (llm.py)
                                                               │
                                                        Streamlit UI
                                                     (streamlit_app.py)
```

**Design Choice — Option B (single collection):** WikiRAG uses one ChromaDB collection (`wiki_rag`) with `entity_type` metadata filtering instead of two separate collections. This simplifies the architecture, enables cross-type queries about both people and places simultaneously, and uses ChromaDB's native `where={"entity_type": "person"}` filtering rather than managing two collection lifecycles.

## Prerequisites

- Python 3.10 or higher
- [Ollama](https://ollama.ai) installed and running
- 8 GB+ RAM (16 GB recommended for comfortable mistral:7b inference)
- ~4 GB disk space for model weights + vector store

## Installation

### 1. Clone the repository
```bash
git clone <repo-url>
cd wiki-rag
```

### 2. Create and activate a virtual environment
```bash
python -m venv .venv

# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate
```

### 3. Install Python dependencies
```bash
pip install -r requirements.txt
```

### 4. Install and start Ollama
```bash
# Install from https://ollama.ai

# Pull the required models
ollama pull mistral
ollama pull nomic-embed-text

# Start the Ollama server (keep this running in a separate terminal)
ollama serve
```

### 5. Ingest Wikipedia data
```bash
python run_ingestion.py
```

This fetches all 40 entities from Wikipedia via the MediaWiki Action API, chunks them, embeds them in parallel with `nomic-embed-text` (8 concurrent requests via `httpx`), and stores them in ChromaDB. Expect ~5–15 minutes depending on your machine. If some entities fail due to Wikipedia rate limiting (HTTP 429), simply re-run the script — already-ingested entities are skipped automatically via the SQLite tracker.

### 6. Start the application
```bash
streamlit run app/streamlit_app.py
```

Open http://localhost:8501 in your browser.

## Design Decisions

### Option B — Single ChromaDB Collection
All documents (people and places) are stored in one `wiki_rag` collection with `entity_type` metadata. The classifier routes queries to filter by type when appropriate. Cross-type queries search both simultaneously and merge results by cosine distance.

### MediaWiki Action API Fetcher
Wikipedia content is fetched via the official MediaWiki Action API (`https://en.wikipedia.org/w/api.php`) with `prop=extracts&explaintext=1`. This avoids the unreliable `wikipedia` Python package and gives clean, plain-text article extracts directly. SSL verification is disabled for environments where the Wikipedia certificate causes issues.

### Sliding Window Chunker
Text is split into 500-character chunks with 100-character overlap. Reference sections (References, Bibliography, Further reading, External links, etc.) are stripped before chunking. Chunks with high density of citation markers (doi, ISBN, archive.org links) are discarded as low quality. Chunks smaller than 100 characters are also discarded.

### Token-level Query Classifier
The query classifier checks full entity names first, then individual name tokens (so "Messi" matches "Lionel Messi", "Einstein" matches "Albert Einstein"). It falls back to keyword scoring and defaults to `"both"` if no signal is found. This is fast, deterministic, and requires no ML model.

### Entity-targeted Retrieval
When a query mentions a known person or place, the retriever fetches chunks filtered specifically to that entity using ChromaDB `where={"entity_name": ...}` metadata queries. This avoids retrieving irrelevant chunks from other entities and dramatically improves answer precision. General similarity search pads the results if the targeted fetch doesn't return enough chunks.

### Parallel Embedding with httpx
Embeddings are generated concurrently using `httpx.AsyncClient` with a semaphore limiting to 8 parallel requests. This gives roughly 6–8× speedup over sequential embedding during ingestion.

### Raw HTTP to Ollama
Both the embedder and LLM caller use `requests`/`httpx` directly to the Ollama REST API (`/api/embeddings` and `/api/generate`). No Ollama Python SDK is used, keeping the code transparent and dependency-light.

## Limitations

- Only covers the 40 hardcoded entities; cannot answer questions outside this scope
- mistral:7b may occasionally misinterpret complex multi-part questions
- No conversation memory — each query is answered independently
- Wikipedia content reflects the state at ingestion time; not live
- Some entities may fail ingestion on first run due to Wikipedia rate limiting; re-running handles this automatically

## Project Structure

```
wiki-rag/
├── ingest/           # Wikipedia fetching (MediaWiki API), chunking, entity lists
├── embeddings/       # Ollama nomic-embed-text via httpx (parallel) + requests (fallback)
├── vectorstore/      # ChromaDB setup and query logic
├── retrieval/        # Query classifier and entity-targeted retriever
├── generation/       # Ollama mistral:7b via raw HTTP
├── database/         # SQLite ingestion tracker
├── app/              # Streamlit GUI (4 pages), CSS dark theme, sidebar quick questions
├── data/             # ChromaDB + SQLite (gitignored)
├── run_ingestion.py  # Standalone ingestion script
└── requirements.txt
```
