# Product Requirements Document — WikiRAG

**Version:** 1.0  
**Status:** Implemented  
**Author:** Local AI Assistant Project  
**Date:** 2026-04-28

---

## 1. Problem Statement

Students and researchers who want to learn about famous historical figures and notable places currently rely on external services (Google, ChatGPT, Wikipedia) that require internet access, send queries to remote servers, and cannot guarantee factual grounding of their responses. There is no offline, privacy-preserving AI assistant that answers encyclopedic questions with traceable sources.

WikiRAG solves this by building a fully local Retrieval-Augmented Generation system that ingests Wikipedia content once and then answers questions entirely on the user's machine — no cloud, no tracking, no hallucinations beyond the source material.

---

## 2. Goals and Non-Goals

### Goals
- Build a fully local RAG pipeline that ingests Wikipedia and answers questions grounded in that content
- Provide a polished, production-quality Streamlit GUI requiring no CLI interaction for end users
- Support queries about famous people and notable places, with cross-type mixed queries
- Track ingestion status persistently so data survives application restarts
- Cite sources for every answer so users can verify claims

### Non-Goals
- Real-time Wikipedia fetching at query time (data is pre-ingested)
- Support for Wikipedia articles outside the 40 hardcoded entities
- Multi-user deployment or authentication
- Fine-tuning or training any ML model
- Mobile or native app

---

## 3. User Stories

**US-01 — Student Researcher:**  
As a student writing a report, I want to ask "What did Marie Curie discover?" and receive a concise, source-cited answer so that I can verify the information came from Wikipedia without searching through full articles.

**US-02 — Curious Learner:**  
As a curious learner, I want to type natural language questions in a chat interface so that I don't need to learn any special query syntax or CLI commands.

**US-03 — Data Administrator:**  
As a system administrator, I want to see which entities are ingested and re-ingest failed ones individually so that I can maintain the knowledge base without re-running the entire pipeline.

**US-04 — Developer / Debugger:**  
As a developer, I want to directly query ChromaDB with raw text and inspect chunk scores so that I can debug retrieval quality and tune chunk parameters.

**US-05 — Privacy-Conscious User:**  
As a privacy-conscious user, I want the entire system to run on localhost with no network calls during inference so that my questions are never sent to external servers.

**US-06 — Offline User:**  
As a user without reliable internet, I want to be able to answer questions after the initial ingestion without any internet connection required.

**US-07 — Power User:**  
As a power user, I want to see which chunks were retrieved for each answer along with their relevance scores so that I can evaluate answer quality myself.

## 4. Functional Requirements

### 4.1 Ingestion
- FR-01: The system must fetch Wikipedia pages for all 20 people and 20 places using the `wikipedia` Python library
- FR-02: Disambiguation errors must be handled by attempting the first suggested alternative
- FR-03: Page-not-found errors must be logged to SQLite as `status='failed'` and must not crash the pipeline
- FR-04: Text must be chunked using a sliding window of 500 characters with 100 characters overlap
- FR-05: Chunks shorter than 100 characters must be discarded
- FR-06: Each chunk must carry metadata: entity name, entity type, chunk index, source URL
- FR-07: Embeddings must be generated using Ollama `nomic-embed-text` via raw HTTP POST
- FR-08: Chunks with embeddings must be upserted into a persistent ChromaDB collection named `wiki_rag`
- FR-09: Every ingestion event must be recorded in SQLite with timestamp, status, chunk count, and URL
- FR-10: Already-ingested entities must be skipped on re-run unless explicitly requested

### 4.2 Query Pipeline
- FR-11: User queries must be embedded using the same `nomic-embed-text` model
- FR-12: Queries must be classified as "person", "place", or "both" using entity name matching and keyword scoring
- FR-13: Classification must use entity name lookup before keyword fallback
- FR-14: Retrieval for "both" queries must merge top-3 person + top-3 place results and deduplicate
- FR-15: Answers must be generated by Ollama `mistral:7b` using a strict, grounded system prompt
- FR-16: When no relevant context exists, the system must return "I don't know based on the available information"

### 4.3 GUI
- FR-17: The application must have exactly four pages: Chat, Ingest Data, Debug/Explore, About
- FR-18: The Chat page must use `st.chat_message` and `st.chat_input`
- FR-19: Each assistant answer must be accompanied by a collapsible source panel showing chunk details
- FR-20: The Ingest page must show real-time progress during ingestion
- FR-21: The Debug page must allow direct ChromaDB search with type filtering and result count control
- FR-22: The sidebar must show live Ollama status, document counts, and a refresh button
- FR-23: Empty query submissions must show a warning and not trigger retrieval

### 4.4 Error Handling
- FR-24: All seven defined error conditions must be handled gracefully with informative user messages
- FR-25: The application must not crash on any single entity failure during batch ingestion

## 5. Non-Functional Requirements

### Performance
- NFR-01: Retrieval (embedding + ChromaDB query) must complete within 5 seconds on typical hardware
- NFR-02: The Streamlit UI must remain responsive during background ingestion operations
- NFR-03: Batch embedding must use a small inter-call delay (50ms) to avoid overwhelming Ollama

### Reliability
- NFR-04: ChromaDB storage must be persistent across application restarts
- NFR-05: SQLite tracker must persist ingestion state across restarts
- NFR-06: The system must handle any single Wikipedia fetch failure without aborting the full pipeline

### Privacy & Locality
- NFR-07: Zero data must leave the user's machine during inference or retrieval
- NFR-08: All models must run locally via Ollama
- NFR-09: No external API keys required

### Code Quality
- NFR-10: No LangChain, LlamaIndex, Haystack, or similar RAG frameworks may be used
- NFR-11: Embedding and LLM calls must use raw `requests` HTTP (no Ollama SDK)
- NFR-12: All chunking and retrieval logic must be custom-implemented

## 6. System Architecture

```
[User] → [Streamlit GUI] → [Retriever]
                               ├── [Classifier] → entity_type filter
                               ├── [Embedder] → query vector
                               └── [ChromaDB] → top-k chunks
                                       ↓
                              [LLM (Ollama mistral:7b)] → answer

[Admin] → [Ingest Page] → [Fetcher] → [Chunker] → [Embedder] → [ChromaDB]
                                                                    ↑
                                                              [SQLite Tracker]
```

**Single Collection Design (Option B):**  
One `wiki_rag` ChromaDB collection stores all documents. Entity type is stored as document metadata and used as a `where` filter at query time. This avoids the complexity of maintaining two separate collection lifecycles and allows the retriever to serve cross-type queries by simply omitting the filter.

## 7. Data Flow

### Ingestion
1. `run_ingestion.py` iterates through `PEOPLE` and `PLACES`
2. For each entity: check SQLite → skip if already ingested
3. Fetch Wikipedia page content via `wikipedia_fetcher.py`
4. Chunk content via `chunker.py` (sliding window, 500 chars, 100 overlap)
5. Batch-embed all chunks via `embedder.py` (Ollama nomic-embed-text)
6. Upsert chunks + embeddings into ChromaDB via `chroma_store.py`
7. Record success/failure in SQLite via `sqlite_tracker.py`

### Query
1. User submits question in Streamlit chat input
2. `classify_query()` determines entity type (person/place/both)
3. `embed_text()` converts query to a vector
4. `chroma_query()` retrieves top-k chunks using cosine similarity + metadata filter
5. `generate_answer()` builds a grounded prompt and calls Ollama mistral:7b
6. Answer and source chunks rendered in Streamlit UI

## 8. UI/UX Requirements

### Chat Page
- Must show an empty state with suggestion chips when no messages exist
- Suggestion chips must pre-fill the chat input when clicked
- Query type classification (person/place/mixed) must be visible in the sidebar
- Sources must be collapsible (not shown by default) to avoid cluttering the view

### Ingest Page
- Already-ingested entities must be visually distinct from pending ones (not shown as checkboxes)
- Progress must update in real time (not just at completion)
- Terminal-style log must show individual pipeline steps (fetch → chunk → embed → store)
- "Re-ingest All" must be visually distinct (red border) to signal destructive intent

### Debug Page
- Distance scores must be color-coded (green = close, amber = medium, red = far)
- Full chunk text must be visible (not truncated) in the expanded view

### Global
- Dark theme ("Dark Intelligence") applied globally via CSS injection
- No default Streamlit elements visible (hamburger menu, footer)
- Custom fonts: Syne (headings), DM Sans (body), JetBrains Mono (code)

## 9. Success Metrics

- All 40 entities successfully ingest on first run on reference hardware
- All 13 specified example queries return sensible, grounded answers
- "President of Mars" and "John Doe" queries return the "I don't know" fallback
- Application loads in under 3 seconds on localhost
- Zero crashes during batch ingestion even if individual entity fetches fail
- No LangChain or LlamaIndex imports detected anywhere in the codebase

---

## 10. Out of Scope

- Adding new entities without code changes
- Conversation memory / multi-turn dialogue
- User authentication or multi-user support
- Production deployment (see recommendation.md)
- Fine-tuning models on the ingested content
- Support for non-English Wikipedia content
- Real-time Wikipedia updates
- PDF, video, or audio ingestion
